# 计划交接指南（阶段 6 路径菜单细则）

> **阅读时机**：阶段 6 展示计划后，用户选择路径 2（与 codex 计划讨论）或路径 3（计划拆分落盘）时阅读本文件；选择路径 1（直接实施）无需加载。
> SKILL.md 阶段 6 定义菜单"是什么"，本文件定义路径 2/3"怎么做"。

---

## 共用目录约定（路径 2/3）

- 位置：使用者项目的 `docs/` 目录下，`docs/YYYYMMDD-计划名/`（如 `docs/20260612-用户通知功能/`）
- 日期取当天；计划名用需求的短语义名，跟随用户项目语言
- `docs/` 不存在时直接创建（`mkdir -p` 幂等）；同日同名冲突时追加序号（`-2`、`-3`）
- 路径 2 的共识计划与路径 3 的三件套落同一目录——先讨论后拆分时，拆分直接基于已落盘的共识 `plan.md` 补齐 `tasks.md` 与 `progress.md`，不另开目录

---

## 路径 2：与 codex 计划讨论

**前置检测**：运行 `codex --version`，失败 → 告知用户 codex CLI 不可用（需安装 OpenAI Codex CLI），重出精简菜单（仅路径 1/3）。

**沙箱选择**：统一使用 `-s read-only`——评审只需读代码上下文，不给写权限可避免两个 agent 同时改盘冲突。

**轮次编排**（上限 3 轮——评审意见通常在 2-3 轮内收敛，超过 3 轮多为偏好拉锯，应交还用户裁决）：

```
plan_file = 把阶段 6 计划全文写入临时文件
            （含需求摘要 / 澄清结论 / 技术设计 / 实施步骤 / 风险）
round 1:
  评审指令 = "你是计划评审者。审阅以下实施计划，逐条指出：风险遗漏、
              步骤缺陷、更优替代方案、过度设计。每条给出
              [同意/异议 + 理由 + 建议改法]。"
  cat 指令与计划 | codex exec -s read-only -o /tmp/codex-review-r1.md -
  # 长计划经 stdin 管道传入（argv 有长度上限）；
  # -o 把最终回复落盘后用 Read 读取，避免解析事件流

每轮结束:
  读 -o 输出 → 整合为三类：[共识项 / 分歧项 / codex 新建议]
  → 向用户展示本轮摘要（一句话级；分歧项注明双方立场）

round 2..3（仅当存在分歧项）:
  response = 对每条分歧的立场与理由（接受则改计划；不接受则给反驳依据）
  codex exec resume --last -o /tmp/codex-review-rN.md "${response}"
  # --last 取最近会话；讨论期间不要并发其他 codex exec，否则 --last 错位
  # round 1 输出如含 session id，可改用 codex exec resume <SESSION_ID> 精确恢复

收敛或达 3 轮上限:
  共识计划草案 = 原计划 + 双方接受的修改
  未决分歧 → 列入"开放问题"节（双方立场并陈，由用户裁决）
```

**失败处理**：单轮建议设置超时（如 300s），超时或命令失败 → 该轮作废，基于已有材料直接出共识草案。讨论轮次不持久化：会话中断后重选路径 2 即重新开始——单轮成本低，断点设计不值得。

**落盘与流向**：共识草案展示给用户 → 用户确认（仍可修改）→ 写入 `docs/【YYYYMMDD-计划名】/plan.md`（结构按 [plan-split-templates.md](../assets/plan-split-templates.md) 的 plan.md 模板，"codex 共识记录"节必填）→ **重出精简菜单（仅路径 1/3）**——此时计划已是共识版，用户可直接实施或继续拆分。

## 路径 3：计划拆分落盘

**产物**：`docs/【YYYYMMDD-计划名】/` 下三件套（模板见 [plan-split-templates.md](../assets/plan-split-templates.md)）：

| 文件 | 职责 | 中断恢复时的角色 |
|------|------|------------------|
| `plan.md` | 计划总览：阶段 1-6 产出（需求/发现/澄清/设计/风险） | 背景输入，恢复时先读 |
| `tasks.md` | 任务清单：每任务一节（编号/状态/依赖/目标/操作步骤/验收标准/产出文件） | 执行单元定义 |
| `progress.md` | 进度面板：当前任务指针、状态总表、恢复指引、变更日志 | 恢复入口，只读此文件即知全局 |

**拆分粒度**：每任务 = 一个可独立验证的最小交付单元（有自己的验收标准），含 1-5 个操作步骤；超过 5 步说明任务过大，继续拆。任务间依赖显式声明（依赖字段），无依赖的任务标注可并行。

**任务状态机**：`pending → in_progress → completed`；任何状态可转 `blocked`（备注必须写明阻塞原因与解除条件），解除后回到原状态。

**生成流程**：

1. 基于阶段 6 计划（或路径 2 的共识计划）拆分任务，生成三件套
2. 向用户展示拆分结果（任务列表 + 粒度说明）并确认后续，二者合并为一次交互：
   - **立即开始实施** → 将 tasks.md 任务逐项注册进运行时任务管理，进入阶段 7，执行单位 = tasks.md 的任务
   - **本会话到此为止** → 输出恢复指引（见"恢复自包含性"），阶段 7-9 留待后续会话
   - 用户对拆分本身有调整意见 → 修改三件套后重新确认

**worktree 隔离实施（强制）**：路径 3「立即开始实施」进入阶段 7 时，实施须在独立的 git worktree 中进行，不直接在主工作区写代码——与"计划拆分落盘"本身的可追溯、可回滚、可中断恢复定位一致（实施失败或计划废弃可整体丢弃 worktree 而不污染主工作区）。

- **创建时机**：进入阶段 7（立即实施）时基于当前分支创建 worktree，分支名含计划目录名（如 `plan/20260612-用户通知功能`），并在 `progress.md` 变更日志记录 worktree 路径与分支名；「本会话到此为止」不创建，留待后续会话首次实施时建。会话恢复时 worktree 已存在则复用、不存在则补建。
- **每任务提交（强制）**：每完成一个任务，在 worktree 分支上 commit 本次任务的代码变更（message 对齐任务编号，如 `feat(T3): xxx`），随后执行双轨同步（更新 `tasks.md` 状态与 `progress.md` 的指针/总表/变更日志，再同步运行时任务状态）。顺序固定：**commit 代码 → spec 自检（见下条）→ 更新文档 → 同步运行时**——让每个任务在 git 与文档中都有独立、可回溯的锚点，per-task 审查与未来 D 的 reviewer 可基于任务 commit 的 SHA 区间核对。降级：`git rev-parse --is-inside-work-tree` 失败（非 git 仓库）→ 跳过 commit，仅更新文档并在变更日志注明"未 commit"及原因；worktree 退主工作区等其他场景仍正常 commit。
- **每任务 spec 自检（强制）**：commit 后、文档同步前，主线程（不派子代理）对照 `T{n}` 的验收标准（tasks.md）重读本任务 diff，仅查两件阶段8 不覆盖的事：(a) **over/under-building**——是否写了任务没要求的代码，或漏了任务要求的产出文件/验收点；(b) **契约锚定**——本任务确立的契约（函数签名/数据结构/API 形态）是否可能被后续任务隐式重新解释。发现即就地修正并补一次 commit（或 amend），再继续文档同步。**禁止重复阶段8 的 A/B/C**——主线程不在此猎 bug/查风格/查规范（那是阶段8 的活，自检重复只造噪音与虚假安全感）。light 档可省略（单文件/加字段类任务 over/under-building 风险极低）。
- **并行任务群**：tasks.md 中依赖字段为「无」的任务子集若并行推进，各起独立子 worktree（或独立分支）避免文件写冲突；带依赖序的任务链在同一 worktree 内按序推进。
- **合并与清理**：全部任务完成并通过整体代码审查（见下方「跨会话的审查纪律」）后，worktree 分支合并回来源分支并删除 worktree；审查发现问题在 worktree 内修复后重审。
- **降级**：`git rev-parse --is-inside-work-tree` 失败（非 git 仓库）、当前分支受保护且用户未授权开分支、或 Codex 沙箱无 worktree 写权限 → 退回主工作区实施，并在 `progress.md` 变更日志记录"未 worktree 隔离"及原因。
- **不引入外部依赖**：用原生 `git worktree add`，不调用 superpowers 的 using-git-worktrees（ra 保持零外部 skill 依赖）。

**运行时任务注册（双轨同步）**：

| 环境 | 注册 | 状态同步 |
|------|------|----------|
| Claude Code | 每任务一条 `TaskCreate`（subject 用 `T{n}: 任务名` 对齐编号） | `TaskUpdate` |
| Codex | `update_plan` 每任务一项（同样 `T{n}: 任务名` 命名） | `update_plan` |

同步规则：**文档为持久层，运行时任务为会话视图**——每完成一个任务，先 commit 本次变更（见上方「worktree 隔离实施」的每任务提交），再更新 `tasks.md` 状态字段与 `progress.md`（指针、总表、变更日志），最后同步运行时任务状态。顺序固定的理由：会话可能随时中断，文档落后于运行时会让恢复点失真；反向最多造成运行时视图短暂滞后，无害。

**恢复自包含性**：恢复指引固定文案写在 `progress.md` 内，任何 agent（不限于本 skill 触发的会话）读到该文件都能照做继续——下次会话用户只需说"继续 `docs/【目录名】` 的计划"。新会话恢复时按 `progress.md` 重建运行时任务列表：已 `completed` 的任务不再注册。

**跨会话的审查纪律**：分会话执行时，本 skill 的阶段 8 只覆盖当前会话的变更；`progress.md` 恢复指引第 5 条规定"全部任务完成后对整体变更运行一次代码审查"（强制步骤，非可选提示）。

## per-task spec 门的 opt-in 触发

路径 3 实施默认**不派 per-task spec 子代理门**——主线程 spec 自检（见上方「worktree 隔离实施」的每任务 spec 自检）已以零子代理成本捕获 spec 合规的主要收益（over/under-building + 契约锚定）。仅当观察到以下信号时，建议 opt-in 启用 per-task spec 门（每任务 commit 后派 1 个 code-reviewer，限 `BASE_SHA..HEAD_SHA` 区间，对照 tasks.md 验收标准查 spec 合规，confirmed spec 类高严重 findings==0 才放行）：

- **跨任务契约错被合规化放大**：阶段8 confirmed 发现的根因可追溯至 >1 个任务前已完成的任务（T1 的契约错被 T2/T3 忠实实现，阶段8 因"实现符合可见契约"而漏判）；
- **契约面覆盖截断**：completeness critic 的覆盖声明标记了契约面的 coverage_note 截断；
- **deep 档契约定义任务**：deep 档任务链含跨多文件的契约定义任务（为后续任务定义接口/数据结构），错误跨文件传播返工成本高，优先评估 opt-in。

三个信号都在阶段8 现有产物（confirmed 报告 + completeness critic 覆盖声明）里，无需新增遥测。opt-in 启用前需配套：review-findings schema 加 `spec` category、code-reviewer 加"维度 D: spec 合规"（仅编排传入 tasks.md 验收标准时激活，否则维持 A/B/C 默认，避免影响共享 agent 的其他消费者）、standard/deep 默认开 light 关，并先在一个 standard 档真实任务集实测 spec 类高严重 finding 命中率（建议 >5% 才默认开）。
