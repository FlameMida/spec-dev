# 计划交接指南（阶段 6 路径菜单细则）

> **阅读时机**：阶段 6 展示计划后，用户选择路径 2（与 codex 计划讨论）或路径 3（计划拆分落盘）时阅读本文件；选择路径 1（直接实施）无需加载。
> SKILL.md 阶段 6 定义菜单"是什么"，本文件定义路径 2/3"怎么做"。

---

## 与 spec-flow 的边界

满足升级条件（实施跨多个会话、需要正式验收/变更留痕/归档）时，优先建议切换 spec-flow（见 SKILL.md 阶段 6 升级出口）——它有 runtime 状态机、独立验收与归档能力。

路径 3 的定位是**轻量落盘**：纯 markdown、无 runtime 依赖、人可直接读改，适合"要可追溯、可中断恢复，但不想引入 `.specs/` 工作区"的场景。

不要对同一需求同时维护三件套与 spec-flow 的 spec.md/plan.md——两套进度状态必然漂移，以先建立的一套为准。

---

## 共用目录约定（路径 2/3）

- 位置：使用者项目的 `docs/` 目录下，`docs/【YYYYMMDD-计划名】/`（如 `docs/20260612-用户通知功能/`）
- 日期取当天；计划名用需求的短语义名，跟随用户项目语言
- `docs/` 不存在时直接创建（`mkdir -p` 幂等）；同日同名冲突时追加序号（`-2`、`-3`）
- 路径 2 的共识计划与路径 3 的三件套落同一目录——先讨论后拆分时，拆分直接基于已落盘的共识 `plan.md` 补齐 `tasks.md` 与 `progress.md`，不另开目录

## 路径 2：与 codex 计划讨论

**前置检测**：运行 `codex --version`，失败 → 告知用户 codex CLI 不可用（需安装 OpenAI Codex CLI），重出精简菜单（仅路径 1/3）。

**沙箱选择**：统一使用 `-s read-only`——评审只需读代码上下文，不给写权限可避免两个 agent 同时改盘冲突。

**轮次编排**（上限 3 轮——评审意见通常在 2-3 轮内收敛，超过 3 轮多为偏好拉锯，应交还用户裁决）：

```
plan_file = 把阶段 6 计划全文写入临时文件
            （含需求摘要 / 澄清结论 / 技术设计 / 实施步骤 / 风险）
round 1:
  评审指令 = "你是计划评审者。审阅以下实施计划，逐条指出：风险遗漏、
              步骤缺陷、更优替代方案、过度设计。每条给出
              [同意/异议 + 理由 + 建议改法]。"
  cat 指令与计划 | codex exec -s read-only -o /tmp/codex-review-r1.md -
  # 长计划经 stdin 管道传入（argv 有长度上限）；
  # -o 把最终回复落盘后用 Read 读取，避免解析事件流

每轮结束:
  读 -o 输出 → 整合为三类：[共识项 / 分歧项 / codex 新建议]
  → 向用户展示本轮摘要（一句话级；分歧项注明双方立场）

round 2..3（仅当存在分歧项）:
  response = 对每条分歧的立场与理由（接受则改计划；不接受则给反驳依据）
  codex exec resume --last -o /tmp/codex-review-rN.md "${response}"
  # --last 取最近会话；讨论期间不要并发其他 codex exec，否则 --last 错位
  # round 1 输出如含 session id，可改用 codex exec resume <SESSION_ID> 精确恢复

收敛或达 3 轮上限:
  共识计划草案 = 原计划 + 双方接受的修改
  未决分歧 → 列入"开放问题"节（双方立场并陈，由用户裁决）
```

**失败处理**：单轮建议设置超时（如 300s），超时或命令失败 → 该轮作废，基于已有材料直接出共识草案。讨论轮次不持久化：会话中断后重选路径 2 即重新开始——单轮成本低，断点设计不值得。

**落盘与流向**：共识草案展示给用户 → 用户确认（仍可修改）→ 写入 `docs/【YYYYMMDD-计划名】/plan.md`（结构按 [plan-split-templates.md](../assets/plan-split-templates.md) 的 plan.md 模板，"codex 共识记录"节必填）→ **重出精简菜单（仅路径 1/3）**——此时计划已是共识版，用户可直接实施或继续拆分。

## 路径 3：计划拆分落盘

**产物**：`docs/【YYYYMMDD-计划名】/` 下三件套（模板见 [plan-split-templates.md](../assets/plan-split-templates.md)）：

| 文件 | 职责 | 中断恢复时的角色 |
|------|------|------------------|
| `plan.md` | 计划总览：阶段 1-6 产出（需求/发现/澄清/设计/风险） | 背景输入，恢复时先读 |
| `tasks.md` | 任务清单：每任务一节（编号/状态/依赖/目标/操作步骤/验收标准/产出文件） | 执行单元定义 |
| `progress.md` | 进度面板：当前任务指针、状态总表、恢复指引、变更日志 | 恢复入口，只读此文件即知全局 |

**拆分粒度**：每任务 = 一个可独立验证的最小交付单元（有自己的验收标准），含 1-5 个操作步骤；超过 5 步说明任务过大，继续拆。任务间依赖显式声明（依赖字段），无依赖的任务标注可并行。

**任务状态机**：`pending → in_progress → completed`；任何状态可转 `blocked`（备注必须写明阻塞原因与解除条件），解除后回到原状态。

**生成流程**：

1. 基于阶段 6 计划（或路径 2 的共识计划）拆分任务，生成三件套
2. 向用户展示拆分结果（任务列表 + 粒度说明）并确认后续，二者合并为一次交互：
   - **立即开始实施** → 将 tasks.md 任务逐项注册进运行时任务管理，进入阶段 7，执行单位 = tasks.md 的任务
   - **本会话到此为止** → 输出恢复指引（见"恢复自包含性"），阶段 7-9 留待后续会话
   - 用户对拆分本身有调整意见 → 修改三件套后重新确认

**运行时任务注册（双轨同步）**：

| 环境 | 注册 | 状态同步 |
|------|------|----------|
| Claude Code | 每任务一条 `TaskCreate`（subject 用 `T{n}: 任务名` 对齐编号） | `TaskUpdate` |
| Codex | `update_plan` 每任务一项（同样 `T{n}: 任务名` 命名） | `update_plan` |

同步规则：**文档为持久层，运行时任务为会话视图**——每完成一个任务，先更新 `tasks.md` 状态字段与 `progress.md`（指针、总表、变更日志），再同步运行时任务状态。顺序固定的理由：会话可能随时中断，文档落后于运行时会让恢复点失真；反向最多造成运行时视图短暂滞后，无害。

**恢复自包含性**：恢复指引固定文案写在 `progress.md` 内，任何 agent（不限于本 skill 触发的会话）读到该文件都能照做继续——下次会话用户只需说"继续 `docs/【目录名】` 的计划"。新会话恢复时按 `progress.md` 重建运行时任务列表：已 `completed` 的任务不再注册。

**跨会话的审查纪律**：分会话执行时，本 skill 的阶段 8 只覆盖当前会话的变更；`progress.md` 的恢复指引提示"全部任务完成后对整体变更运行一次代码审查"。
