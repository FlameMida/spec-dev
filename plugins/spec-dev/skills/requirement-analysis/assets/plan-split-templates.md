# 计划拆分三件套模板

> 路径 3（计划拆分落盘）生成全部三件；路径 2（codex 计划讨论）的共识计划只使用 plan.md 模板。
> 生成规则、拆分粒度与双轨同步见 [plan-handoff.md](../references/plan-handoff.md)。

## plan.md 模板

```markdown
# 【计划名】实施计划

> 创建：YYYY-MM-DD ｜ 来源：requirement-analysis 阶段 6 ｜ 状态：planning

<!-- 状态取值：planning | in_progress | completed | abandoned -->

## 需求摘要
[阶段 1 产出：核心功能、业务目标、功能边界]

## 关键发现
[阶段 2-3 产出：相关文件、现有模式、外部研究结论——精简到影响实施的条目]

## 澄清结论
[阶段 4 产出：用户确认的取舍、约束、待防守假设]

## 技术设计
[阶段 5 产出：数据结构 / API / 服务层设计]

## 实施步骤总览
[编号步骤列表；已拆分时每步标注 tasks.md 任务编号，如「1. xxx → T1-T3」]

## 风险与注意事项
[风险及缓解措施]

## codex 共识记录（经路径 2 时必填，未经讨论则删除本节）
- 讨论轮次：N
- 关键分歧与结论：[分歧点 → 采纳立场及理由]
- 开放问题：[未决分歧，双方立场并陈，留用户裁决]
```

## tasks.md 模板

```markdown
# 任务清单

> 状态机：pending → in_progress → completed；任何状态可转 blocked（备注必须写明阻塞原因与解除条件），解除后回到原状态。
> 每任务一个可独立验证的最小交付单元，操作步骤用 checkbox 跟踪到步。

## T1: 【任务名】

| 字段 | 值 |
|------|-----|
| 状态 | pending |
| 依赖 | 无（或 T-id 列表，如 T2、T3） |
| 对应计划步骤 | 步骤 1 |
| 产出文件 | path/to/file |

**目标**：[一句话——这个任务交付什么]

**操作步骤**：
- [ ] 步骤 1
- [ ] 步骤 2

**验收标准**：[可执行的验证命令或可观察的结果]

**备注**：[blocked 原因、执行中的发现等；无则留空]

## T2: 【任务名】

[同 T1 结构]
```

## progress.md 模板

```markdown
# 进度面板

## 状态
- 当前任务：T1（无进行中任务时写「未开始」或「全部完成」）
- 总进度：0/N 已完成
- 最后更新：YYYY-MM-DD HH:mm
- 最后会话摘要：[一句话——上次做到哪、有什么未决事项]

## 任务状态总表

| 任务 | 名称 | 状态 | 完成时间 |
|------|------|------|----------|
| T1 | xxx | pending | — |
| T2 | xxx | pending | — |

## 会话恢复指引（固定文案，请勿删改）

1. 读同目录 `plan.md` 了解需求背景与设计
2. 查上方状态面板与总表，定位当前任务
3. 读 `tasks.md` 对应任务节，按操作步骤执行
4. 每完成一个任务：先更新 `tasks.md` 状态字段与本文件（指针、总表、变更日志），再同步运行时任务状态（Claude Code 用 `TaskUpdate` / Codex 用 `update_plan`）
5. 全部任务完成后：把 `plan.md` 状态改为 completed，并对整体变更运行一次代码审查

## 变更日志（append-only）

| 时间 | 事件 | 说明 |
|------|------|------|
| YYYY-MM-DD HH:mm | 计划创建 | 由 requirement-analysis 阶段 6 生成 |
```
