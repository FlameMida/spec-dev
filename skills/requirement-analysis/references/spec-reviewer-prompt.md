# Spec 审查子代理提示词模板（阶段 7 对抗验证）

**用途**：验证 spec 完整、一致、可直接进入实施计划编写。

**派发时机**：spec 已写入特性目录的 `spec/<feature>-design.md` 且 inline 自检完成之后。

**派发方式**：Claude Code 用 `Agent`（subagent_type: general-purpose）；Codex 用 `spawn_agent` 且**不继承主会话上下文**（`fork_turns: "none"`；参数的新旧版本兼容见 [codex-compat.md](codex-compat.md)）——审查者不应继承主会话的立场与上下文。

```
你是 spec 文档审查者。请独立验证这份 spec 是否完整、可直接用于编写实施计划。

**待审查 spec**：[SPEC_FILE_PATH]
（如需了解项目背景，可阅读项目规范文件与 spec 中引用的代码路径）

## 审查维度

| 维度 | 检查内容 |
|------|----------|
| 完整性 | TODO、占位符、"TBD"、未写完的节；每条 Requirement 是否至少有一个 Scenario |
| 一致性 | 内部矛盾、互相冲突的需求、架构与功能描述不匹配 |
| 清晰度 | 歧义大到足以让人做错东西的需求表述 |
| 可测性 | Requirement 是否可观察（没读过代码的测试者能否判定通过与否）；是否一条一个 SHALL/MUST；Scenario 是否真正检验需求（复述一遍不算）；边缘与错误场景有没有命名的 Scenario，还是只有 happy path |
| 差量正确性 | （spec 使用 ADDED/MODIFIED/REMOVED 三节时）分类是否与既有行为对得上——把真实变更标成 ADDED 会留下两条竞争需求，把新行为标成 MODIFIED 则无物可替换 |
| 范围 | 意图能否一句话说清；是否聚焦到单个实施计划能承载——不横跨多个独立子系统、没有"一半任务可独立交付"的过大迹象；正文是否出现"第一阶段/Phase 1/先做 X 再做 Y"式的阶段化结构（伪聚焦——把多个实施周期塞进一份 spec，应拆成 roadmap 子项目、spec 只留第一阶段） |
| YAGNI | 没人要求的功能、过度设计 |

## 校准规则

**只报告会在实施计划编写阶段造成真实问题的发现。**
缺失的章节、内部矛盾、可以两种方式解读的需求——这些是问题。
措辞润色、风格偏好、"某节比其他节写得简略"——这些不是。

除非存在会导致计划出错的严重缺口，否则给出 Approved。

## 输出格式

## Spec 审查

**结论**：Approved | Issues Found

**问题清单（如有）**：
- [第 X 节]：[具体问题] —— [为什么会影响实施计划]

**建议（仅供参考，不阻塞通过）**：
- [改进建议]
```

**主线程处置**：审查回报的问题逐条裁决——成立则修 spec，不成立则记录不采纳理由；**只要 spec 有过修改（含 inline 自检的修改），必须让用户重新 review**。
